# 🕐 Linux Kernel Device Driver Clock


리눅스 커널 공간(Kernel Space)에서 직접 하드웨어를 제어하는 **통합 디바이스 드라이버 기반 임베디드 시계 시스템**입니다.  
별도의 유저 공간(User Space) 프로그램 없이, 커널 드라이버 자체적으로 시간 설정 및 UI 제어가 가능합니다.

---

## 1. 개요

DS1302 RTC 모듈을 통해 정확한 시간을 유지하고, I2C OLED 디스플레이에 현재 시간을 시각화하며,  
별도의 유저 공간 프로그램 없이 커널 드라이버 자체적으로 시간 설정 및 UI 제어 가능한 임베디드 시계 시스템 구축

| 항목 | 내용 |
|------|------|
| **수행 기간** | 2025.12.24 ~ 2025.12.29 |
| **사용 기술** | C, Device Driver, I2C, Workqueue |
| **담당 역할** | RTC(DS1302) 모듈, 로터리 엔코더 기능 구현 (디바이스 드라이버) |

---

## 2. 구성도 (System Architecture)

- **Raspberry Pi 4 :** 메인 제어 플랫폼, 커널 드라이버 실행 환경
- **DS1302 (RTC 모듈) :** GPIO 기반 3-wire 프로토콜로 정확한 시간 유지. 전원이 꺼져도 배터리로 시간 보존
- **SSD1306 (OLED 모듈) :** I2C 프로토콜(Address: `0x3C`)로 연결, 현재 시간 및 UI 시각화
- **EC11 (로터리 엔코더) :** 인터럽트 기반으로 회전/버튼 입력 처리, FSM 상태 전환 트리거

---

## 3. 기능 설명

### FSM 기반 UI 제어

초기 부팅 시 선택 모드로 시작하며, 로터리 엔코더의 회전과 버튼 클릭을 통해 두 가지 모드 간 전환이 이루어집니다.

1. **선택 모드 :** 로터리 엔코더 회전으로 수정할 항목(Year/Month/Day/시/분/초)에 포커스 이동. 버튼 클릭 시 수정 모드로 전환
2. **수정 모드 :** 로터리 엔코더 회전으로 현재 포커스된 항목의 값 증가/감소. 변경 즉시 DS1302 RTC에 하드웨어 반영. 버튼 클릭(롱 프레스) 시 저장 후 선택 모드로 복귀

```
시스템 시작
    │
    ▼
┌─────────────┐   버튼 클릭   ┌─────────────┐
│  선택 모드   │ ────────────▶ │  수정 모드   │
│ OLED Display│               │ OLED Display│
│ (회전: 이동) │ ◀──────────── │ (회전: 증감) │
└─────────────┘   버튼 클릭   └─────────────┘
                  (저장 및 복귀)
```

### 인터럽트 + Workqueue 기반 비동기 처리

인터럽트 핸들러(ISR)에서 I2C 통신 같이 `sleep`이 필요한 작업은 직접 수행할 수 없습니다.  
이를 해결하기 위해 **Workqueue**를 사용하여 OLED 화면 갱신 작업을 프로세스 문맥으로 위임합니다.

1. **인터럽트 발생 :** 로터리 엔코더 회전 시 GPIO 인터럽트(ISR) 즉시 감지
2. **작업 위임 :** ISR에서는 회전 방향 계산 및 시간 변수 변경만 수행 후 Workqueue에 화면 갱신 예약
3. **비동기 처리 :** Workqueue Worker가 I2C 통신을 통해 OLED 화면 갱신 및 DS1302 Write 처리

```
Rotary Encoder 회전
        │
        ▼
   [Interrupt ISR]
   - 회전 방향 계산
   - 시간 변수 변경
   - Workqueue에 작업 예약  ──────▶  [Workqueue Worker]
   - 즉시 종료                        - I2C 통신 (OLED 갱신)
                                      - DS1302 RTC Write
```

### RTC(DS1302) 동기화

1. **시간 유지 :** 전원이 꺼져도 배터리 백업으로 시간이 유지되며 재부팅 시 자동으로 현재 시간 로드
2. **즉각 반영 :** 수정 모드에서 값 변경 시 즉시 DS1302 레지스터에 기록
3. **타이머 인터럽트 :** 주기적으로 RTC에서 시간을 읽어 OLED 화면을 자동 갱신

## 4. 주요 기술

| 구성 요소 | 역할 및 기술 |
| --- | --- |
| **Raspberry Pi 4** | 커널 드라이버 실행 환경, I2C-1 Bus Master |
| **DS1302 (RTC)** | GPIO 3-wire 프로토콜, `udelay` 기반 비트뱅잉, 시간 데이터 BCD 인코딩 |
| **SSD1306 (OLED)** | I2C 서브시스템(`i2c_master_send`), 128x32 픽셀 직접 제어, 폰트 스케일링 |
| **EC11 (로터리 엔코더)** | GPIO 인터럽트(`request_irq`), A/B 상 엣지 검출로 회전 방향 판별 |
| **FSM** | 선택 모드 ↔ 수정 모드 상태 전환, 포커스 기반 UI 제어 |
| **Workqueue** | 인터럽트 문맥에서 처리 불가한 I2C 작업을 프로세스 문맥으로 지연 처리 |
| **Interrupt** | 비동기 입력 처리, 타이머 인터럽트를 통한 화면 주기적 동기화 |

### Workqueue
리눅스 커널의 지연 처리 기법으로, 인터럽트 문맥에서 처리하기 어려운 긴 작업이나 대기가 필요한 작업을 프로세스 문맥으로 미루어 처리합니다. 커널 스레드 환경에서 실행되므로 I2C 통신 등 대기 함수(`sleep`) 사용이 가능합니다.

### I2C (Inter-Integrated Circuit)
SDA, SCL 두 개의 선만 사용하는 직렬 통신 프로토콜로, Master-Slave 구조에서 마스터가 통신을 주도합니다. 각 장치는 고유 주소를 가지므로 하나의 버스에 여러 장치를 연결할 수 있으며, 리눅스 커널의 I2C 서브시스템(`i2c_master_send`, `i2c_master_recv`)을 활용합니다.

### Interrupt
CPU가 실행 중 예기치 않은 이벤트 발생 시 현재 작업을 중단하고 즉시 처리하는 메커니즘입니다. 로터리 엔코더 회전/버튼 입력과 타이머 인터럽트 기반 화면 동기화에 활용되며, **ISR 내에서는 처리가 빠르게 끝나야 하고 `sleep` 함수 사용이 불가**합니다.

## 5. 트러블 슈팅 (Troubleshooting)

**[커널 모듈 로드 시 라즈베리 파이 꺼짐]**

- **현상 :** 커널 모듈을 로드하면 라즈베리 파이가 꺼지는 현상 발생
- **원인 :** 인터럽트 핸들러 내에서 `sleep` 함수를 호출하여 커널이 패닉을 일으키며 종료
- **해결 :** Workqueue로 `sleep`이 필요한 작업(I2C 통신)을 프로세스 문맥으로 이동하여 해결

**[OLED 화면 노이즈]**

- **현상 :** OLED 화면에 노이즈가 발생하여 정상적으로 출력되지 않는 상태
- **원인 :** 다른 테스트 드라이버가 이미 I2C 드라이버 자원을 점유하고 있어 OLED 드라이버 초기화 실패
- **해결 :** 시스템 재부팅을 통해 자원 점유 상태를 초기화하여 해결

**[RTC 비정상적인 날짜/시간 출력]**

- **현상 :** `165-165-165 165:165:165` 등 비정상적인 값이 지속적으로 출력
- **원인 :** 하드웨어 배선 불량(점퍼 와이어 접촉 불량)으로 추정
- **해결 :** RTC 점퍼 와이어를 새로 교체하여 물리적 접촉 불량 문제 해결

## 6. 개발 후기

**1. 커널 레벨 제약사항 체득**  
인터럽트 핸들러 내에서 `sleep` 사용이 불가하다는 제약을 직접 커널 패닉으로 경험하면서, 커널 문맥(Context)의 중요성을 몸소 깨달았습니다. Workqueue를 통해 인터럽트 문맥과 프로세스 문맥을 분리하는 설계 패턴을 적용하며 리눅스 커널의 지연 처리 메커니즘을 깊이 이해하게 되었습니다.

**2. Rust 커널 드라이버 도전**  
C 기반 드라이버 외에 Rust로도 동일한 기능을 구현해보며, 커널 Rust API의 현재 한계(GPIO, Timer 관련 API 미완성)를 직접 체감했습니다. C 커널과 다른 관점의 추상화 방식을 비교하면서 Rust 커널 개발자들이 추구하는 메모리 안전성 철학과 그 과정의 트레이드오프를 이해할 수 있었습니다.
